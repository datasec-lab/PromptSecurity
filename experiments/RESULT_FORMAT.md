# Experiment Result Fields and Explanations

- **index**  
  The sequential number of this experiment sample (usually starting from 0 or 1), used to uniquely identify each result.

- **dataset_name**  
  The name of the dataset (e.g., 'harmbench', 'airbench'), indicating which dataset this sample comes from.

- **sample_index**  
  The index of the test sample in the original dataset.

- **clean_prompt**  
  The original (unattacked/undefended) prompt text from the dataset.

- **target_llm_name**  
  The name of the target large language model (LLM), such as 'gpt-4o', 'Llama-3.1-8B-Instruct', etc.

- **target_llm_type**  
  The type of the target LLM: 'api' (for API calls) or 'local' (for local models).

- **target_llm_parameters**  
  The configuration parameters for the target LLM (as dict/json), such as temperature, max_tokens, top_p, etc. (if any).

- **llm_response_on_clean**  
  The raw response generated by the target LLM for the clean_prompt.

- **llm_response_time_clean**  
  The time taken by the target LLM to generate a response to the clean_prompt (in seconds, float).

- **judger_name**  
  The name(s) of the Judger(s) (can be one or more, stored as a list), such as 'gpt_judger_contextual_harmbench'.

- **judger_config**  
  The configuration parameters for the Judger (dict/json), such as prompts, thresholds, used model, etc.

- **judger_result_on_clean**  
  The judgement result(s) from the Judger on the response to the clean_prompt (e.g., SAFE, UNSAFE, LABEL, etc.; can be a single value or a list).

---

- **attack_method**  
  Name of the attack method (e.g., 'AutoDAN', 'PromptInjection', or 'no' to indicate no attack).

- **attack_type_I**  
  Attack type I: 'black-box' or 'white-box'.

- **attack_type_II**  
  Attack type II: whether an assistant LLM is required ('no' if not, otherwise the name of the assistant LLM, e.g., 'gpt-3.5-turbo').

- **attack_config**  
  Attack method parameters (dict/json).

- **attacked_prompt**  
  The prompt text after the attack, used for testing.

- **attack_runtime**  
  The total time taken to generate the adversarial sample (in seconds, float).

- **attack_query_count**  
  The number of queries sent to the target LLM during the attack.

- **assistant_llm_query_count**  
  The number of queries made to the assistant LLM during the attack (0 if none).

- **llm_response_on_attacked**  
  The target LLM��s response to the attacked_prompt.

- **judger_result_on_attack**  
  Judger��s judgement result(s) on the response to the attacked prompt (can be a single value or a list).

---

- **defense_method**  
  Name of the defense method (e.g., 'input_filter', 'output_filter', 'model_filter', or 'no' for no defense).

- **defense_type_I**  
  Defense type I: 'input_filter', 'model_filter', or 'output_filter'.

- **defense_type_II**  
  Defense type II: whether gradients/logits are required ('yes'/'no').

- **defense_type_III**  
  Defense type III: whether an assistant LLM is required ('no' or the name of the assistant LLM).

- **defense_config**  
  Defense method parameters (dict/json).

- **llm_response_on_clean_under_defense**  
  Target LLM��s response to clean_prompt under defense.

- **judger_result_on_clean_under_defense**  
  Judger��s judgement result on the defended clean_prompt response.

- **llm_response_on_attacked_under_defense**  
  Target LLM��s response to attacked_prompt under defense.

- **llm_response_time_under_defense**  
  The time taken for the LLM to respond under defense (in seconds, float).

- **judger_result_on_attack_under_defense**  
  Judger��s judgement result(s) on the defended attacked_prompt response.

---

- **experiment_name**  
  Name or identifier of the experiment (e.g., 'phase1_harmbench_no_no_gpt-4o_multi_judger') for traceability.

- **experiment_timestamp**  
  Timestamp of when this sample result was saved (preferably in ISO format or Unix timestamp).

---

# Notes

- All fields are recommended to be saved in the above order, with lowercase and underscores for field names.
- Optional/unused fields should be set to null or an empty string.
- Multi-value fields (such as judger_name, judger_result_on_clean) should be stored as lists.
- All long texts should be saved in full, without truncation (jsonl friendly).

## 4要素实验执行流程

**防御优先模式**: 基于白盒/自适应攻击威胁模型，攻击者了解防御机制并能针对性构造攻击。

**执行流程**:
```
1. 防御模型: 创建defended_model (defense + model)
2. 攻击处理: 针对defended_model进行攻击 → attacked_prompt  
3. 模型响应: defended_model处理attacked_prompt
4. 评判分析: 评估(attacked_prompt, defended_response)
```

**使用场景**: 评估防御机制在最坏情况下的鲁棒性，符合现实对抗环境。

**威胁模型特点**:
- 攻击者具有完整的防御机制知识
- 能够针对特定防御构造精确攻击
- 提供更严格和现实的安全评估

## Attack Target Logic

**Critical Logic for Experiment Design and Result Reuse:**

### Attack Target Determination
- **When defense is present**: The attack targets the **defended model** (defense + model combination)
- **When no defense**: The attack targets the **undefended model** directly

### Practical Implications
This logic has crucial implications for result reuse:

**Example Scenario:**
- Experiment A: `sample + attack_A + defense_A + model + judger`
- Experiment B: `sample + attack_A + defense_B + model + judger`

**Result Reuse Rules:**
- ✅ **CAN reuse**: `sample + model` clean response (same model, same sample)
- ❌ **CANNOT reuse**: `attack_A` results from Experiment A in Experiment B
- **Reason**: In Experiment A, `attack_A` targets `defense_A + model`, while in Experiment B, `attack_A` should target `defense_B + model`

### Component Dependencies
The correct dependency chain is:
```
Sample → Model(Clean) → Attack(+Defense if present) → Defense → Judger
```

Each component depends on **ALL** upstream components:
- **Model response**: Depends on model + sample
- **Attack result**: Depends on model + sample + attack + defense (if present)
- **Defense result**: Depends on model + sample + attack + defense  
- **Judger result**: Depends on model + sample + attack + defense + judger

### Cache Key Construction
To ensure correct reuse behavior, cache keys must include all dependencies:
- **Model**: `{model}|{dataset}|{sample}`
- **Attack**: `{model}|{attack}|{defense}|{dataset}|{sample}`
- **Defense**: `{model}|{attack}|{defense}|{dataset}|{sample}`
- **Judger**: `{model}|{attack}|{defense}|{judger}|{dataset}|{sample}`

**Note**: Include `"no_attack"` or `"no_defense"` explicitly when components are not used.
